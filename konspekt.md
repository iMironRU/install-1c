# Конспект `install-1c`

Конституция репозитория. Семь инвариантов, на которых держится всё остальное.

Любое изменение в репо, нарушающее один из инвариантов, требует либо отката, либо явного ADR с пересмотром и переподписанием.

---

## 1. Установка производна от цели

Без зафиксированного «зачем» процесс бессмысленный.

Каждый сценарий имеет:

- имя на трёх языках (см. инвариант 5),
- целевое состояние,
- критерий проверки.

Сценарий без цели и критерия — не сценарий, а заметка.

## 2. Источник один

Все артефакты (скрипты для Windows / Linux / macOS, сайт-инструкция) — производные.

Любое изменение «как» начинается с правки источника в `src/`, никогда — с правки скриптов в `dist/` или текста сайта.

`dist/` находится в `.gitignore` и пересобирается из источника.

## 3. Технический и дидактический слои валидируются отдельно

Зелёный тест скрипта не означает понятную инструкцию.

- Технический слой: скрипт работает, постусловия выполняются. См. `tests/technical/`.
- Дидактический слой: текст полон, согласован, проходит ревью на читателе. См. `tests/didactic/`.

Один зелёный не вытягивает второй. Оба обязательны для статуса `published`.

## 4. Контракт шага

У каждого атомарного шага три обязательных секции:

- **preconditions** — что должно быть истинно до начала.
- **action** — что именно выполняется.
- **postconditions** — исполняемая проверка результата.

Постусловие исполняемо: команда, возвращающая 0 или ожидаемый вывод. «Открылось окно» — не постусловие. «Файл такого-то размера лежит по такому-то пути» — постусловие.

## 5. Три языка перевода

Каждый сценарий и каждый шаг описаны на трёх языках:

- **user** — в лексике пользователя («хочу научиться разрабатывать», «установите программу»).
- **business** — в терминах задачи и роли («АРМ разработчика», «рабочее место с тонким клиентом», «серверная установка»).
- **machine** — точно: имена бинарей, путей, ключей реестра, команд.

Если хотя бы один из трёх отсутствует — артефакт не готов.

## 6. Лицензирование — отдельный слой

Установка платформы и активация лицензии не смешиваются.

У лицензирования собственный поток с той же структурой: что нужно → как получить → как активировать → как проверить. Запутывать эти потоки в одних и тех же шагах запрещено.

## 7. Источники только официальные

Дистрибутивы и обновления — `users.v8.1c.ru`, `1c.ru`. Всё остальное вне scope.

Альтернативные сборки упоминаются только в `didactic_notes.pitfalls` как «здесь не идём — почему». Скрипты не содержат ссылок на неофициальные источники.

## 8. Шаг — функция с явной сигнатурой

Переиспользование через вызов, не через копирование.

Каждый шаг имеет:

- **parameters** — именованные входы с типом и дефолтом. Параметр появляется только когда реально вызывается с разными значениями из разных мест. Иначе это лишняя гибкость без потребителя.
- **returns** — именованные выходы, которые следующие шаги могут потребить через `bind`.
- **тело** — preconditions / action / postconditions с подстановкой `{{param-name}}`. Полиморфизм по ОС — внутри тела, не параметр.

Путь (`src/paths/`) — упорядоченный список вызовов (`call`) с аргументами (`args`) и привязкой результатов к локальным переменным (`bind`). Путь — интеграционный тест сценария. Шаг — юнит.

Дидактика расщепляется надвое:
- `didactic_notes.why` в определении шага — общее «зачем шаг существует».
- `call_notes.why` в вызове пути — «зачем именно здесь, с этими аргументами».

При переиспользовании шага в разных сценариях оба текста различаются. Одинаковый `call_notes.why` в двух путях — признак переабстракции.
