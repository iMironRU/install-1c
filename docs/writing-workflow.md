# Процесс написания (SDD)

Книга пишется по методологии Spec-Driven Development:
**сначала определяем что создаём, потом создаём**.

Изменение контента всегда начинается с изменения спецификации.

---

## Этапы жизни книги

### 1. Конституция (`spec/constitution.md`)

Прежде чем написать первый параграф, ответьте:
- Что эта книга ЕСТЬ и чем НЕ ЯВЛЯЕТСЯ?
- Какие принципы нельзя нарушать?
- Кто читатель, в одной фразе?

Конституция — это ограничитель. Если идея противоречит конституции,
она не попадает в книгу, даже если кажется полезной.

### 2. Спецификация (`spec/specification.md`)

После конституции — полное описание продукта:
- Кто читатель в деталях
- Что он умеет на входе и на выходе
- Объём, структура, метрики

### 3. Планирование модуля

Перед написанием модуля:
```
./book.sh status           ← посмотреть текущее состояние
/book-plan модуль-N        ← спланировать параграфы с Claude Code
```

Зафиксируйте план в `spec/decisions.md`.

### 4. Написание параграфов

```
/book-write §N.N [тема]    ← написать параграф
```

Статус нового параграфа: `draft`.

### 5. Самопроверка

```
/book-analyze §N.N         ← проверить параграф
```

После исправлений меняйте статус: `draft` → `review`.

### 6. Финализация

Когда параграф прошёл проверку: `review` → `ready`.

Только `ready`-параграфы попадают в релизные сборки.

### 7. Релиз

```
./book.sh release
```

Скрипт спросит уровень (patch/minor/major) и записи в changelog.
После `git push && git push --tags` GitHub Actions соберёт все форматы.

---

## Статусы параграфов

| Статус | Значение | В релизе? | На сайте? |
|--------|----------|-----------|-----------|
| `draft` | Написан, не проверен | Нет (только ночные) | Да, с баннером |
| `review` | На проверке | Да (настройка) | Да, с иконкой 🔍 |
| `ready` | Финальный | Да | Да |

---

## Правила changelog

Что **попадает** в changelog:
- Новая глава или модуль
- Принципиальное исправление в примере кода
- Значительная переработка раздела
- Исправление фактической ошибки

Что **не попадает**:
- Опечатки и запятые
- Стилистические правки
- Реструктуризация без смены смысла

Changelog пишет **автор**, не скрипт.
